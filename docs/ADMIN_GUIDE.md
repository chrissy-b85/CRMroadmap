## 1. User Management (Auth0)

### 1.1 Creating a staff account

1. Log in to the [Auth0 dashboard](https://manage.auth0.com/).
2. Navigate to **User Management → Users**.
3. Click **Create User**.
4. Enter the staff member's email address, set a temporary password, and select
   the **Username-Password-Authentication** connection.
5. Assign the `staff` role under **Roles**.
6. The user will receive a welcome email. Ask them to change their password on
   first login.

### 1.2 Creating a participant account

Participant accounts are created from within the CRM:
1. Open the participant detail page in the CRM.
2. Under **Portal Access**, click **Create Portal Account**.
3. The system creates an Auth0 user with the `participant` role and sends an
   invitation email.

### 1.3 Resetting a password

Navigate to **User Management → Users**, find the user, and click
**Send Password Reset Email**.

### 1.4 Disabling an account

Find the user in Auth0 and toggle **Block User**. This prevents login without
deleting the account.

### 1.5 MFA configuration

MFA is enforced by an Auth0 Action for all `staff` role users. To manage the
MFA policy, go to **Security → Multi-factor Auth** in the Auth0 dashboard.

## 2. Xero Integration Setup

### 2.1 Create a Xero app

1. Log in to [Xero Developer](https://developer.xero.com/).
2. Create a new **OAuth 2.0** app.
3. Set the redirect URI to
   `https://api.ndis-crm.your-org.com.au/xero/callback`.
4. Note the **Client ID** and **Client Secret**.

### 2.2 Store credentials in Secret Manager

```bash
echo -n "YOUR_XERO_CLIENT_ID" | gcloud secrets versions add xero-client-id --data-file=-
echo -n "YOUR_XERO_CLIENT_SECRET" | gcloud secrets versions add xero-client-secret --data-file=-
```

### 2.3 Authorise the connection

1. Open the CRM and navigate to **Settings → Integrations → Xero**.
2. Click **Connect Xero**.
3. You will be redirected to Xero's OAuth login. Approve the permissions.
4. The CRM will store the refresh token and begin syncing.

### 2.4 Webhook setup (optional)

Xero webhooks allow the CRM to be notified of payment events. In the Xero
Developer portal, add a webhook endpoint pointing to
`https://api.ndis-crm.your-org.com.au/xero/webhook` and store the webhook
key:

```bash
echo -n "YOUR_WEBHOOK_KEY" | gcloud secrets versions add xero-webhook-key --data-file=-
```

## 3. Microsoft Graph API Setup

### 3.1 Register an Azure AD application

1. Log in to the [Azure portal](https://portal.azure.com/).
2. Navigate to **Azure Active Directory → App registrations → New registration**.
3. Set the name to `NDIS CRM`.
4. Note the **Application (client) ID** and **Directory (tenant) ID**.
5. Under **Certificates & secrets**, create a new client secret and note the
   value immediately.

### 3.2 Grant API permissions

Under **API permissions**, add the following **Application** permissions
(not Delegated) from **Microsoft Graph**:

| Permission | Purpose |
|-----------|---------|
| `Mail.ReadWrite` | Read and move emails from the shared mailbox |
| `Mail.Send` | Send notification emails |

Click **Grant admin consent** to activate the permissions.

### 3.3 Store credentials in Secret Manager

```bash
echo -n "YOUR_CLIENT_ID"     | gcloud secrets versions add msgraph-client-id     --data-file=-
echo -n "YOUR_CLIENT_SECRET"  | gcloud secrets versions add msgraph-client-secret  --data-file=-
echo -n "YOUR_TENANT_ID"      | gcloud secrets versions add msgraph-tenant-id      --data-file=-
```

## 4. GCS Bucket Management

### 4.1 Bucket overview

| Bucket | Purpose |
|--------|---------|
| `ndis-crm-prod-documents` | NDIS plans, invoices, statements, service agreements |
| `ndis-crm-prod-backups` | Cloud SQL backup exports |

### 4.2 Generating a signed URL for a document

Staff access documents via time-limited signed URLs generated by the backend.
The backend service account has `roles/storage.objectViewer` on the documents
bucket. URLs expire after 15 minutes.

### 4.3 Manual file operations

```bash
# List objects
gsutil ls gs://ndis-crm-prod-documents/

# Download a file
gsutil cp gs://ndis-crm-prod-documents/participants/123/plan.pdf /tmp/

# Delete a file (use with caution)
gsutil rm gs://ndis-crm-prod-documents/participants/123/plan.pdf
```

### 4.4 Lifecycle policy

Documents bucket: archived versions are deleted after 90 days, keeping the 3
most recent versions. Backups bucket: all objects deleted after 90 days. These
policies are managed by Terraform in `infra/terraform/gcs.tf`.

## 5. Celery Task Monitoring

Celery workers process background tasks including:
- Polling the Outlook mailbox for new invoices
- Running Document AI OCR
- Pushing approved invoices to Xero
- Sending budget alert notifications
- Generating monthly PDF statements

### 5.1 Cloud Run logs

View worker logs in Google Cloud Logging:

```bash
gcloud logging read "resource.type=cloud_run_revision AND resource.labels.service_name=ndis-crm-worker" \
  --limit=100 --format=json
```

Or use the [GCP Console Logs Explorer](https://console.cloud.google.com/logs/).

### 5.2 Checking task queues

Connect to the Redis instance (via Cloud SQL Auth proxy or an internal VM) and
inspect the Celery queue:

```bash
redis-cli -h 10.x.x.x LLEN celery
```

### 5.3 Retrying failed tasks

Failed tasks can be retried from the CRM admin interface or by triggering the
relevant Celery task directly via the backend API with an admin token.

## 6. Database Backups

### 6.1 Automated backups

Cloud SQL is configured with:
- Automated daily backups at 2 AM AEST
- Point-in-time recovery (PITR) enabled
- 30-day backup retention
- 7-day transaction log retention

Backups are managed by Cloud SQL and visible in the
[GCP Console → SQL → Backups](https://console.cloud.google.com/sql/instances).

### 6.2 Manual export

To create a manual export to the backups GCS bucket:

```bash
gcloud sql export sql ndis-crm-postgres \
  gs://ndis-crm-prod-backups/manual/$(date +%Y%m%d-%H%M%S).sql.gz \
  --database=ndis_crm
```

### 6.3 Restoring from backup

To restore the database to a specific point in time:

1. In the GCP Console, navigate to **SQL → ndis-crm-postgres → Backups**.
2. Select the backup or choose **Point-in-time recovery**.
3. Follow the prompts. **This will overwrite the current database.**
4. After restoration, run `alembic upgrade head` to ensure the schema is
   up to date.

## 7. Incident Response

### 7.1 Service is down

1. Check [GCP Console → Cloud Run](https://console.cloud.google.com/run) for
   the service status.
2. Review recent logs for errors.
3. If a bad deployment caused the issue, roll back using:
   ```bash
   gcloud run services update-traffic ndis-crm-backend \
     --to-revisions REVISION_ID=100 \
     --region australia-southeast1
   ```
4. Check Cloud SQL and Redis are healthy.

### 7.2 Data breach suspected

1. Immediately revoke all active sessions by rotating the Auth0 application
   secret.
2. Disable the relevant GCP service accounts.
3. Contact the organisation's privacy officer and follow the OAIC data breach
   notification procedure.
4. Preserve all logs for forensic investigation.

### 7.3 Xero sync failed

1. Navigate to **Settings → Integrations → Xero** in the CRM.
2. If the OAuth token has expired, click **Reconnect Xero** and re-authorise.
3. Invoices with `Failed` Xero status can be retried from the invoice detail
   page.
